Guia de implementação do sistema sonoro de paragens

Este guia explica como ativar os avisos de "próxima paragem" e "paragem atual" para passageiros, incluindo equipamentos recomendados, endpoints da API e checklist de entrada em produção.

1) Arquitetura recomendada

  1. App/API: calcula estado de anúncio por serviço e evita repetições.
  2. Dispositivo de bordo: reproduz áudio TTS e consulta a API em ciclo curto.
  3. GPS: usa pontos da app do motorista (e/ou tracker) para localizar a viatura.
  4. GTFS: usa sequência de paragens para identificar próxima/atual.

2) Equipamentos compatíveis

Opção A (piloto rápido e económico)

  • Tablet Android 10"+ com 4G/Wi-Fi.
  • Coluna ativa 12V ou amplificador compacto.
  • Conversor 24V/12V estabilizado para alimentação.

Opção B (mais robusta)

  • Raspberry Pi 4/5 + DAC + amplificador classe D.
  • Armazenamento industrial e fonte veicular.
  • Caixa com proteção para vibração e calor.

Em ambas as opções, o software de bordo pode consumir os mesmos endpoints da API já implementados.

3) Endpoints da API para anúncios

3.1 Obter estado de anúncio

GET /services/:serviceId/announcement-state?preannounceMeters=300&arrivalMeters=60
Authorization: Bearer <token>

Resposta inclui: nextStop, currentStop, distância/ETA, e triggers announceNextStop / announceArrivedStop.

3.2 Confirmar anúncio tocado

POST /services/:serviceId/announcement-mark
Authorization: Bearer <token>
Content-Type: application/json

{
  "stopId": "1234",
  "stopSequence": 17,
  "announcementType": "next"
}

Use announcementType=arrived quando tocar o aviso de chegada à paragem.

4) Fluxo operacional no dispositivo de bordo

  1. Identificar o serviceId ativo da viatura.
  2. Fazer polling do endpoint de estado (tipicamente a cada 2-4 segundos).
  3. Se trigger "next" estiver ativo, tocar "Próxima paragem: ...".
  4. Se trigger "arrived" estiver ativo, tocar "Paragem: ...".
  5. Após tocar, chamar /announcement-mark para registo anti-repetição.

5) Configuração recomendada de thresholds

  • Pré-aviso: 250-350m (urbano) ou 350-500m (vias rápidas).
  • Chegada: 40-80m.
  • Polling: 2s a 4s.

Ajuste por linha e contexto de tráfego. O objetivo é evitar avisos demasiado cedo ou em duplicado.

6) Fallback offline (já suportado no frontend do motorista)

  1. Se a API falhar repetidamente, o modo local é ativado automaticamente.
  2. O cálculo passa a usar GTFS local + GPS do dispositivo.
  3. Quando a API recuperar, o modo API é restabelecido de forma automática.
  4. O badge visual mostra o modo atual: API, LOCAL ou OFF.

7) Checklist de entrada em produção

  1. Confirmar que os serviços iniciam corretamente com GTFS associado.
  2. Testar 1 linha piloto em horário real por pelo menos 3 dias.
  3. Ajustar volume e voz com feedback de passageiros/motoristas.
  4. Validar comportamento em túneis/zonas de fraco sinal GPS.
  5. Registar procedimento operacional para suporte e manutenção.

8) Teste rápido (fim-a-fim)

  1. Iniciar serviço na app de motorista.
  2. Abrir "Percurso e operações" e ativar anúncios automáticos.
  3. Confirmar mudança de badge para API (ou LOCAL em fallback).
  4. Verificar reprodução em "Próxima paragem" e "Paragem".
  5. Confirmar ausência de repetição indevida na mesma paragem.
