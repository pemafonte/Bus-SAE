Guia completo de integracao Teltonika

Este guia mostra como ligar trackers Teltonika a esta aplicacao por webhook HTTP e por TCP direto (Codec8), incluindo checklist de rede, testes externos e migracao de servidor.

1) Visao geral da arquitetura

  1. App/API: recebe eventos GPS e associa ao servico em curso pela frota/viatura mapeada ao IMEI.
  2. Modo A - Webhook HTTP: gateway (Flespi/Traccar) encaminha JSON para /integrations/teltonika/events.
  3. Modo B - TCP direto: tracker liga por TCP, envia IMEI + Codec8 e recebe ACK.
  4. Persistencia: os pontos entram em service_points com point_source='tracker'.

2) Variaveis de ambiente (backend/.env)

TELTONIKA_WEBHOOK_TOKEN=defina_um_token_forte
TELTONIKA_TCP_PORT=5027
TELTONIKA_TCP_DEBUG=true
PORT=4000

Notas: sem TELTONIKA_TCP_PORT o servidor TCP nao sobe. Em producao, pode deixar TELTONIKA_TCP_DEBUG=false para reduzir logs.

3) Preparacao de rede para ambiente externo

  1. Garanta que o servidor e alcancavel externamente (IP publico ou dominio).
  2. Abra a porta HTTP/HTTPS da API para webhook (ex.: 443 via reverse proxy).
  3. Abra a porta TCP configurada em TELTONIKA_TCP_PORT (ex.: 5027).
  4. Se existir NAT, configure port forwarding para o host da API.
  5. Valide firewall local e firewall cloud (Security Group/NSG).

4) Configuracao por webhook HTTP (gateway)

Use este modo quando os trackers enviam para Flespi/Traccar e o gateway chama a API.

POST /integrations/teltonika/events
Authorization: Bearer <TELTONIKA_WEBHOOK_TOKEN>
Content-Type: application/json

{
  "imei": "356307042441013",
  "lat": 38.7223,
  "lng": -9.1393,
  "timestamp": "2026-04-22T10:00:00Z",
  "speed": 28,
  "heading": 180,
  "fleetNumber": "BUS-101"
}

5) Configuracao por TCP direto (Codec8)

  1. No configurador Teltonika, defina servidor IP/DNS e porta TCP da API.
  2. O dispositivo envia IMEI no handshake; servidor responde 0x01 quando aceite.
  3. Em seguida o dispositivo envia AVL Codec8; servidor responde ACK (4 bytes, numero de records aceites).
  4. Se ACK for 0, verifique logs e mapeamento IMEI/frota na app.

6) Mapeamento obrigatorio para gravar pontos

  1. Na aba "Integracoes GPS", registe o IMEI com a mesma frota usada no inicio do servico.
  2. Garanta que existe um servico in_progress para essa frota.
  3. Sem mapeamento ativo ou sem servico em curso, os eventos serao rejeitados.

7) Testes externos (checklist rapido)

  1. Teste conectividade: GET /health a partir de uma rede externa.
  2. Teste webhook com curl/Postman e valide accepted > 0.
  3. Teste TCP com tracker real e confirme ACK positivo.
  4. No supervisor, confirme que surgem pontos GPS no servico em curso.

8) Migracao para novo servidor (implicacoes)

Sim, ha implicacoes operacionais, mas o desenvolvimento feito continua valido. Ao migrar, precisa atualizar endpoints e conectividade:

  1. Atualizar DNS/IP/porta configurados nos trackers e/ou no gateway.
  2. Atualizar firewall/NAT no novo ambiente.
  3. Copiar variaveis .env (token e porta TCP) para o novo servidor.
  4. Garantir que a base de dados e dados de mapeamento IMEI foram migrados.
  5. Repetir testes de ponta-a-ponta apos cutover.

9) Recomendacoes para evitar retrabalho

  • Use dominio estavel (ex.: gps-api.seudominio.pt) para nao reprogramar trackers a cada troca de IP.
  • Mantenha a porta TCP final o mais estavel possivel entre ambientes.
  • Versione e documente o ficheiro .env por ambiente.
  • Rode teste de validacao apos cada deploy com payload de referencia.
